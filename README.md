# Azure Data Factory ETL Framework
### Metadata-Driven Data Ingestion, Orchestration, and Transformation using Azure Data Factory
---
## Key Highlights
* **Automated File Ingestion:** Built Azure Data Factory (ADF) pipelines to automate the ingestion of sales data from external HTTP sources and local raw landing zones.
* **Metadata-Driven Orchestration:** Implemented dynamic folder scanning and loop routing using Get Metadata, ForEach, If Condition, and Pipeline Variable activities.
* **Parameterized Datasets:** Designed reusable dataset models with runtime parameters for dynamic source and destination mapping.
* **Visual Transformations:** Created Mapping Data Flows with column pruning, filtering, conditional routing, and group-by aggregations.
* **Dual-Trigger Configurations:** Automated pipeline execution using event-based (blob storage creation) and schedule-based triggers.
* **Git Integration:** Linked configurations directly with source control using the `adf_publish` branch for release management.

## 1. Project Overview
This portfolio project is a practical implementation of an end-to-end data ingestion, staging, and transformation framework built inside **Azure Data Factory V2** and backed by **Azure Data Lake Storage (ADLS) Gen2**. 
The repository contains the exact configuration blueprints (JSON templates) for the pipelines, datasets, linked services, and mapping data flows. It provides a modular approach to handling file arrivals, moving them based on filename pattern matching, and cleaning up source folders.
---
## 2. Solution Architecture
The architecture represents an industry-standard, multi-zone landing pipeline inside ADLS Gen2. 
![Architecture Diagram](<img width="1024" height="1024" alt="image" src="https://github.com/user-attachments/assets/dcfeca5a-a51f-4ced-b25a-0b2e3b167416" />
)
The framework organizes files into three logical zones:
1. **Source Zone (Raw Landing):** Receives external file uploads (e.g., `Fact_Sales_1.csv`).
2. **Destination Zone (Staging):** Holds copies of ingested raw files and external HTTP feeds for validation and filtering.
3. **Reporting Zone (Curated):** Stores sanitized data and aggregated summaries generated by Mapping Data Flows.
---
## 3. Business Problem
An organization receives daily transaction and sales sheets from internal uploads and external Git partners. The data needs to be:
* Ingested reliably and archived from the landing zone immediately upon success to avoid double processing.
* Filtered dynamically so that only specific files (e.g., files starting with `Fact`) are forwarded.
* Sanitized to exclude test accounts and scrubbed to prevent PII exposure (such as credit card numbers) in reporting datasets.
* Partitioned and aggregated by payment methods for final audit.
This framework solves these needs by providing a low-maintenance, serverless ingestion and cleansing pipeline.
---
## 4. Solution Overview
The workflow begins when a file is either uploaded to the raw landing zone or fetched from a public GitHub repository. 
* Trigger mechanisms initiate pipelines that copy datasets to staging.
* In staging, a metadata scanner reads the directory, loops through files, filters for files beginning with `Fact`, and copies them to the reporting zone using parameterization.
* Finally, a Mapping Data Flow runs, executing transformations (such as column selection, row filtering, card brand splitting, and maximum product calculation) before storing the final files in ADLS Gen2 reporting folders.

## 9. Data Flows
### Data Flow Design
![Data Flow](images/dataflow.png)
### **[Transform_csv.json](Data%20Pipelines/dataflow/Transform_csv.json)** (Mapping Data Flow)
This is a visual transformation flow that parses incoming transaction datasets:
1. **Source (`sourceCSV`):** Reads CSV records from the reporting zone filesystem.
2. **Select Column (`selectCols`):** Prunes columns to optimize memory usage, removing card numbers, cost, and loyalty flags. Keeps only: `transaction_id`, `transactional_date`, `product_id`, `customer_id`, `payment`, and `price`.
3. **Filter Row (`filter1`):** Discards testing/dummy entries where `customer_id == 12` (`customer_id != 12`).
4. **Conditional Split (`split1`):** Routes records into streams based on payment type:
   * **Visa Stream:** Evaluates `payment == 'visa'`.
   * **Mastercard Stream:** Evaluates `payment == 'mastercard'`.
   * **Amex Stream:** Catch-all default stream.
5. **Derived Column (`derivedColumn`) [Amex Stream]:** Replaces empty or null columns with a default string: `payment = coalesce(payment, 'N/A')`.
6. **Aggregate (`aggregate1`) [Visa Stream]:** Groups transactions to calculate sales statistics: `product_id = max(product_id)`.
7. **Alter Row (`alterRow1`) [Visa Stream]:** Applies row marking rules (`insertIf(1==1)`) to verify output operations.
8. **Sink (`sink`):** Writes the final output to `reporting/dataflowOutput/`.

## 10. Pipelines
The framework relies on four specific pipelines to orchestrate processing steps:
### A. **[pipeline Git.json](Data%20Pipelines/pipeline/pipeline%20Git.json)**
* **Purpose:** Pulls an external sales file from GitHub.
* **Activities:** 
  * `Copy Git` (Copy Data): Copies the raw file from GitHub into `destination/csvfiles/file2.csv` on Azure Data Lake Storage.
### B. **[pipelinemanager.json](Data%20Pipelines/pipeline/pipelinemanager.json)**
* **Purpose:** Performs file ingestion and raw landing clean-up.
* **Activities:**
  * `Copy CSV` (Copy Data): Copies `Fact_Sales_1.csv` from raw `source/csvfiles/` filesystem to staging `destination/csvfiles/` filesystem.
  * `DeleteFile` (Delete Activity): Triggered only after the copy succeeds. It deletes the source file in `source/csvfiles/Fact_Sales_1.csv` to ensure raw landing zone cleanliness and prevent reprocessing loop errors.
### C. **[VarPipeline.json](Data%20Pipelines/pipeline/VarPipeline.json)**
* **Purpose:** Demonstrates parameterization techniques by mapping folder structure to variables.
* **Activities:**
  * `Get Metadata` (Get Metadata): Audits the directory contents of `destination/csvfiles/` (fetches `childItems`).
  * `StoreFiles` (Set Variable): Populates a pipeline array variable `var_files` with the folder's child item list.
### D. **[OnlySelectedFiles.json](Data%20Pipelines/pipeline/OnlySelectedFiles.json)**
* **Purpose:** The main orchestrator pipeline containing conditional looping and transformation triggers.
* **Activities:**
  1. `Get Metadata1` (Get Metadata): Gathers filenames in `destination/csvfiles/`.
  2. `ForEachCSV` (ForEach Loop): Loops sequentially (`"isSequential": true`) over the file list.
     * **Nested `IfFileMatches` (If Condition):** Evaluates if the filename starts with the prefix `"Fact"`: `@startswith(item().name, 'Fact')`.
       * **Nested `Copy data1` (Copy Data) [If True]:** Passes the filename dynamically to parameterized datasets to copy matched files from staging (`destination/csvfiles/`) to reporting (`reporting/csvfiles/`).
  3. `Data Transformatiom` (Execute Data Flow): Triggered after the loop succeeds. It executes the Mapping Data Flow `Transform_csv` using a general compute engine.

## 11. Trigger Orchestration
Trigger configuration profiles control automated execution:
### Trigger Configuration
![ADF Triggers](images/adf_triggers.png)
1. **[MainTrigger.json](Data%20Pipelines/trigger/MainTrigger.json)** (Blob Events Trigger, Status: **Active**):
   * Fires on event: `Microsoft.Storage.BlobCreated`
   * Target Path: `/source/blobs/csvfiles/Fact_Sales_1.csv`
   * Storage Account: `storagedevanshi`
   * Action: Instantly executes `pipelinemanager` when a file lands in the raw container.
2. **[managerTRigger.json](Data%20Pipelines/trigger/managerTRigger.json)** (Blob Events Trigger, Status: **Stopped**):
   * Fires on event: `Microsoft.Storage.BlobCreated`
   * Target Path: `/source/blobs/csvfiles/Fact_Sales1.csv`
   * Action: Standby trigger to monitor alternative raw file names.
3. **[selectedFilesTrigger.json](Data%20Pipelines/trigger/selectedFilesTrigger.json)** (Schedule Trigger, Status: **Active**):
   * Recurrence: Every 15 minutes.
   * Timezone: India Standard Time (IST).
   * Action: Periodically invokes the orchestrator `OnlySelectedFiles` to process newly staged items.

Transformation Phase (Compute-Driven) | | - 'ExecuteDataFlow' starts the 'Transform_csv' mapping data flow. | | - Transaction records load, selecting required columns and filtering out customer 12. | | - Visa transactions are grouped and aggregated to find the max product ID. | | - Cleaned output writes to 'reporting/dataflowOutput/'. | +--------------------------------------------------------------------------------------------------+
